AMEVA Architecture

클라우드 스트리밍과 로컬-퍼스트(Local-First) 패러다임의 결합에 대한 기술적 고찰

1. System Architecture Overview

AMEVA 워크스테이션은 기존 데스크톱 애플리케이션의 한계(무거운 설치, 동기화 지연, 자원 점유)를 극복하기 위해 설계되었습니다. 브라우저 내장 렌더링 엔진과 클라우드 플러그인 스트리밍 방식을 결합하여 Zero-Install 환경을 구현했습니다.

보안이 중요한 API 키와 자격 증명은 서버에 전송하지 않고 OS Keychain API를 통해 데스크톱 물리 계층에 안전하게 암호화되어 보관됩니다.

2. Technical Trade-offs & Fallbacks

복잡한 실시간 협업 시스템을 구축하며 직면한 기술적 한계와 이를 극복하기 위한 Workaround 전략을 표로 정리했습니다.

기술 스택 (Component) 구현 방식 (Approach) 기술적 한계 / Trade-off Workaround & Fallback Strategy
협업 엔진 (Yjs) CRDT 기반 P2P 상태 동기화 참여자가 늘어날수록 State Vector 용량이 기하급수적으로 커져 브라우저 메모리 압박이 생김. 정기적인 Y.encodeStateAsUpdate() 압축 스냅샷을 IndexedDB에 저장하고 불필요한 히스토리는 가비지 컬렉션 처리 (Local-First Fallback).
자격 증명 (Keychain) keytar를 통한 OS 네이티브 암호화 실시간 협업 시 타 유저와 자격 증명(OAuth 토큰 등)이 Yjs 룸을 타고 누출될 보안 위험 존재. [격리 아키텍처] Yjs Document 스키마에서 Credentials 필드를 렌더링 계층에서만 처리하도록 분리하여 네트워크 전송을 물리적으로 차단.
스트리밍 뷰 (Iframe) 플러그인 샌드박싱 로드 Iframe 내부와 외부 간의 이벤트 통신(Drag, Click)이 차단되어 드래그 앤 드롭 리사이징이 끊기는 문제. 리사이징(Drag) 이벤트 발생 순간 Iframe에 pointer-events: none 속성을 주입하고, Drop 시 복구하는 Workaround 적용.
네트워크 (WebRTC) Mesh P2P Audio/Video 사내망/NAT 방화벽 뒤에 있는 유저 간의 P2P 직접 연결(STUN) 실패율 존재. STUN 실패 시 자동으로 TURN 서버로 릴레이(Relay)하도록 Fallback 라우팅 로직 구현 (레이턴시는 소폭 증가하나 연결성 보장).

3. As-Is vs To-Be 패러다임 전환

AMEVA는 사용자의 행동 패턴을 어떻게 변화시키는지 증명합니다.

[As-Is] 파편화된 컨텍스트

작업을 위해 구글 지도, 네이버, 뽀모도로 타이머, 파일 탐색기 등 수많은 앱과 탭을 열어두어야 합니다. 이는 잦은 컨텍스트 스위칭(Context Switching)을 유발하여 생산성을 40% 이상 저하시킵니다.

[To-Be] Unified Workspace (단일 컨텍스트)

AMEVA는 모든 도구를 워크스테이션 내의 모듈형 플러그인으로 흡수합니다. 사용자는 창을 벗어날 필요 없이 마켓플레이스에서 필요한 기능(예: PDF RAG, 칸반 보드)을 원클릭으로 활성화하고 배치할 수 있습니다.
