AMEVA Architecture

클라우드 스트리밍과 로컬-퍼스트(Local-First) 패러다임의 결합에 대한 기술적 고찰

1. Architectural Paradigm: Local-First Cloud Streaming

AMEVA Workstation은 전통적인 데스크톱 애플리케이션의 리소스 제약(Monolithic Resource Hogging)과 웹 애플리케이션의 오프라인 취약성(Network Dependency)을 동시에 해결하기 위해 Local-First Cloud Streaming Architecture를 채택했습니다. 이는 브라우저의 샌드박스 렌더링 엔진(V8 & Blink)을 호스트 OS의 네이티브 리소스와 브릿징하는 하이브리드 접근 방식입니다.

특히 보안 아키텍처에 있어 Zero-Trust 기반의 자격 증명 격리(Credential Isolation)를 구현했습니다. 민감한 API Token 및 OAuth 인증 정보는 클라우드 서버나 브라우저의 로컬 스토리지에 평문으로 저장되지 않으며, 호스트 OS의 Native Keychain API(Windows Credential Manager / macOS Keychain)를 통해 AES-256 규격으로 물리적 암호화되어 관리됩니다.

2. Deep Dive: CRDT 기반 분산 상태 동기화 (Yjs)

실시간 협업(Real-time Collaboration) 기능의 핵심 코어는 Operational Transformation(OT)의 한계를 극복한 CRDT(Conflict-free Replicated Data Type) 알고리즘 기반의 Yjs 엔진입니다. 네트워크 지연(Latency)이나 오프라인 상태에서도 충돌 없는 상태 병합(Deterministic State Merging)을 보장합니다.

  • Vector Clocks & State Vectors: 클라이언트 간의 이벤트 순서를 보장하고 누락된 델타(Delta) 업데이트만 선별적으로 동기화하여 네트워크 페이로드를 최소화합니다.
  • Awareness Protocol: 마우스 커서 위치, 텍스트 Selection 등 비영구적 상태(Ephemeral State)는 CRDT Document에 병합되지 않고 별도의 Awareness 채널을 통해 메모리 레벨에서만 브로드캐스트하여 GC(Garbage Collection) 부하를 방지합니다.

3. Technical Trade-offs & Engineering Resolutions

단일 페이지 애플리케이션(SPA) 내에서 다중 창(Multi-window) 시뮬레이션과 P2P 네트워킹을 구현하는 과정에서 발생한 기술적 병목(Bottlenecks)과 이를 타개한 엔지니어링 의사결정(Engineering Decisions)입니다.

도메인 (Domain) 구현 아키텍처 (Architecture) 기술적 한계 / Trade-off 해결 전략 및 Fallback (Resolution)
State Management CRDT 기반 P2P 상태 동기화 (Yjs) History Append-only 특성으로 인해 시간이 지남에 따라 State Vector의 용량이 선형적으로 증가하여 브라우저의 Heap Memory OOM(Out of Memory) 유발 위험. [State Compaction] 주기적으로 Y.encodeStateAsUpdate()를 호출하여 최적화된 바이너리 스냅샷을 IndexedDB에 압축 저장(Snapshotting)하고, 불필요한 Operation History는 GC를 통해 메모리에서 해제하는 로직 구현.
Security OS Native Keychain 연동 Yjs의 특성상 로컬 State가 클러스터 내 타 노드로 자동 브로드캐스트됨. 사용자 개인의 API Key가 Yjs 룸을 통해 타인에게 유출될 구조적 취약성 존재. [Schema Isolation] 협업을 위한 Shared Schema와 개인화 설정을 위한 Local Schema를 물리적으로 분리. Credentials 데이터는 Yjs Provider 인스턴스에 주입하지 않고 브라우저 Local Context에서만 렌더링되도록 격리(Air-gapped) 아키텍처 적용.
UI/UX (DOM) Iframe 기반 플러그인 샌드박싱 Iframe 내부에 마우스 커서가 진입할 경우 호스트 Window(부모 DOM)가 `mousemove`, `mouseup` 이벤트를 유실하여 윈도우 리사이징/드래깅이 멈추는 UX 단절 현상(Event Swallowing). [Event Tunneling / Pointer Guard] Drag 시작점(mousedown) 이벤트 포착 시 즉각적으로 전체 Iframe에 pointer-events: none CSS 속성을 주입하는 Guard Layer 활성화. Drag 종료(mouseup) 시 복구하여 부드러운 60fps 창 제어 구현.
Network (WebRTC) Mesh Topology P2P Audio Symmetric NAT 또는 엄격한 기업용 방화벽 환경에서 피어 간 직접 연결(STUN) 실패에 따른 통신 단절 문제. [Adaptive ICE Routing] ICE Candidate 수집 과정에서 STUN 연결 실패 Timeout 감지 시, 즉각적으로 TURN(Traversal Using Relays around NAT) 서버로 패킷을 릴레이하는 Fallback 라우팅 로직 구현. 레이턴시 트레이드오프(Latency Trade-off)를 감수하고 Connectivity 99.9% 보장.

4. Paradigm Shift: Unified Context Architecture

AMEVA Workstation의 궁극적인 존재 이유는 파편화된 워크플로우의 통합에 있습니다. 이는 단순한 UI/UX의 개선이 아닌, 작업자의 인지 부하(Cognitive Load)를 줄이는 구조적 혁신입니다.

[As-Is] Fragmented Application Ecosystem (파편화된 생태계)

현대의 지식 근로자는 업무 수행을 위해 브라우저 탭, 슬랙, 노션, 터미널 등 평균 10~15개의 분리된 애플리케이션 창을 전환(Context Switching)하며 사용합니다. 연구 결과에 따르면 이러한 컨텍스트 스위칭은 집중력 회복에 평균 23분을 소모하게 하며 치명적인 생산성 저하를 야기합니다.

[To-Be] AMEVA Unified Micro-frontends (단일 통합 컨텍스트)

AMEVA는 Micro-frontends Architecture를 차용하여 모든 도구를 단일 워크스테이션 내의 독립적인 Plugin Container로 로드합니다. 사용자는 OS의 창을 전환할 필요 없이, 마켓플레이스에서 PDF RAG, Kanban, Python Console 등의 플러그인을 동적으로 메모리에 적재(Lazy-load)하여 무한한 캔버스 위에 자유롭게 배치합니다. 이는 운영체제 위의 운영체제(OS within OS)로써, 데이터와 도구가 하나의 흐름 속에서 상호작용하는 진정한 의미의 Zero-Context Switching 환경을 달성합니다.
